Why Using NonBlockingGame

NonBlockingGames is a Bridges abstraction that allows you to program simple games.

At the core, a game is concerned with four things:

  1. Reading Inputs (which buttons are pressed).
  2. Updating the game state (where is the protagonist).
  3. Rendering to the Screen (what the scren should look like).
  4. Doing it at 30 frame per seconds (getting a smooth experience).
NonBlockingGame in Bridges makes 1., 3., and 4. particularly easy by making some assumptions about the kinds of game one can do. Namely, the NonBlockingGame only enables games to be on a small grid (up to 1024 cells), to use only a few sprites (up to 256 predefined sprites), and to use only a few keys (up to 10 keys).

The benefits is that within these constraints, writing a game is very easy. We will look at how it works in three steps: 1. getting something running; 2. displaying something on the screen; 3. capturing user input.

1. Getting Started

A game in Bridges is creating by creating a new class for your game that derive from NonBlockingGame. Your class has to construct the underlying NonBlockingGame object by passing the usual Bridges credentials, the assignment number, and the size of the game board. We will use a 10x10 board as an example. Then the game object need to be instanciated and the game started.

Minimally, you create a game by writing two functions:

  1. initialize() is called once when the game originally start. It is typically used to set up all the game logic in its initial state. Think for instance about setting up a chess board at the beginning of the game.
  2. gameLoop() is called once per frame of the game. This is the function that you will write to update the game at each frame based on the player's input.
